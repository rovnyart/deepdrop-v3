# DeepDrop AI System Design

## AI Principles

1. AI assists; the user controls execution.
2. Read-only is the default.
3. Prompt safety is not enough; enforce policy in code.
4. Send the smallest useful context.
5. Make AI actions inspectable and reversible where possible.

## User Settings

Required settings:

- OpenAI API key.
- Model.
- Enable AI features.
- Allow schema metadata to be sent.
- Allow sampled row data to be sent.
- Allow mutation queries generated by AI.
- Allow AI to execute read-only queries in chat.
- Retain AI audit logs.

Defaults:

- AI disabled until API key is set.
- Schema sharing enabled only after clear consent during AI setup.
- Sampled row data disabled by default.
- Mutation queries disabled by default.
- AI may propose SQL, but cannot execute mutation SQL by default.

## Feature Set

### Explain Query

Input:

- Selected SQL or active statement.
- Optional schema context.

Output:

- Plain-language explanation.
- Tables/columns involved.
- Potential performance notes.
- Risk classification.

No database execution required.

### AI Query Builder

Input:

- User natural language request.
- Database catalog subset.
- Current editor context.

Output:

```json
{
  "sql": "select ...",
  "explanation": "string",
  "assumptions": ["string"],
  "requiresMutationPermission": false,
  "riskLevel": "low|medium|high"
}
```

Behavior:

- Insert into editor as draft.
- Never auto-run by default.
- Validate and classify before insertion/execution.

### Chat With Database

Capabilities:

- Answer schema questions.
- Generate SQL.
- Run app-approved read-only queries if enabled.
- Summarize small result sets.

Tool boundary:

- AI can request tools.
- App validates tool arguments.
- App executes only allowed tools.
- App returns summarized/redacted outputs where possible.

Example tools:

```json
{
  "name": "list_schemas",
  "description": "List schemas available in the connected database."
}
```

```json
{
  "name": "run_readonly_sql",
  "description": "Run a SQL query that has been validated as read-only by the app.",
  "parameters": {
    "type": "object",
    "properties": {
      "sql": { "type": "string" },
      "maxRows": { "type": "integer", "minimum": 1, "maximum": 200 }
    },
    "required": ["sql", "maxRows"],
    "additionalProperties": false
  }
}
```

### AI Autocomplete

Behavior:

- Only when API key exists and feature is enabled.
- Debounced.
- Uses current editor prefix, schema context, and minimal surrounding text.
- Suggestions appear as ghost text or completion items.
- Never sends result data.

## Safety Policy

### Statement Classes

- `readOnly`: `SELECT`, `WITH` that resolves to read-only, `SHOW`, safe metadata queries.
- `mutation`: `INSERT`, `UPDATE`, `DELETE`, `MERGE`, `TRUNCATE`, `COPY FROM`, data-modifying CTEs.
- `ddl`: `CREATE`, `ALTER`, `DROP`, `COMMENT`, `REINDEX`, `CLUSTER`.
- `admin`: `GRANT`, `REVOKE`, `VACUUM`, `ANALYZE`, `SET ROLE`, security-sensitive operations.
- `unknown`: anything parser cannot confidently classify.

Default AI execution allowed:

- `readOnly` only.

Blocked by default:

- `mutation`, `ddl`, `admin`, `unknown`.

Even when mutation permission is enabled:

- Require explicit user confirmation for execution.
- Show exact SQL.
- Prefer transaction and rollback preview when possible.
- For production-marked connections, require stronger confirmation.

### Enforcement Layers

1. System prompt says AI must not perform mutation unless allowed.
2. Tool descriptions restrict behavior.
3. Structured outputs require a risk field.
4. App SQL classifier validates generated/requested SQL.
5. Execution engine blocks disallowed operations.
6. UI asks for confirmation when policy requires it.

Layer 4 and 5 are mandatory. Prompt-only safety is unacceptable.

## Prompting Patterns

System prompt themes:

- You are assisting inside a PostgreSQL client.
- Prefer correct, minimal SQL.
- Ask clarifying questions when schema is insufficient.
- Do not invent table or column names.
- Do not request or execute mutation queries unless mutation permission is enabled.
- Never include secrets in output.

Developer prompt should include:

- Current policy flags.
- Available tools.
- Output schema.
- Context limits.
- Connected database dialect: PostgreSQL.

User context should include:

- User request.
- Current SQL selection, if relevant.
- Schema subset.
- Optional result summary, only when needed and allowed.

## Structured Outputs

Use OpenAI Structured Outputs or tool calling with strict schemas for:

- Generated SQL.
- Query explanations.
- Risk classifications.
- Chat tool calls.
- Error repair suggestions.

Validate returned JSON in app code. If validation fails, retry once with a corrective prompt or show a controlled error.

## Context Minimization

Prefer sending:

- Schema names.
- Table names.
- Column names/types.
- Constraints/indexes relevant to the prompt.
- Current SQL.
- Error message.

Avoid sending by default:

- Raw row data.
- Credentials.
- Full connection strings.
- Large query history.
- Production data samples.

When sampled rows are needed:

- Ask user or require setting.
- Limit rows and columns.
- Redact likely secrets: emails, tokens, passwords, auth headers, phone numbers where feasible.

## Auditability

For each AI action, optionally store:

- Timestamp.
- Connection ID.
- Feature used.
- Model.
- Input context categories, not necessarily full content.
- Generated SQL.
- Tool calls.
- Policy decisions.
- Execution result summary.

Do not store:

- API key.
- Database password.
- Raw tool response rows unless user opts in.

## Failure Modes

- Missing API key: show setup affordance.
- Invalid API key: settings error and retry.
- Network failure: non-blocking toast/message.
- Invalid SQL generated: show draft plus validation error.
- Policy blocked: explain exactly which setting/policy blocked it.
- Ambiguous schema: ask follow-up or offer assumptions.
